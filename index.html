Process and Thread in Operating System

Introdaction

  1. Process, 即进程。
    • 计算机上所有可运行的软件，包括操作系统，被组织成若干顺序进程，简称进程。
    • 一个进程就是一个正在执行程序的实例，包括程序计数器，寄存器和变量值，每个进程都有一个虚拟 CPU。
    • 单个处理器通过调度算法决定何时停止一个进程的工作，转而为另一个进程服务。
    • 同一段程序运行两遍算作两个进程，但是它们共享代码，只有一个副本内存。
    • 子进程由父进程调用 system call - fork 来创建，两个进程拥有相同的内存映像、同样的环境字符串和同样的打开文件，通常子进程会执行 execve 或其他 system call 来修改其内存映像。
    • 进程有三种状态：运行、阻塞和就绪。程序因为等待输入而阻塞，调度程序选择另一个进程会使进程由运行变为就绪，出现有效输入会使得阻塞变为就绪。
  2. Thread, 即线程。
    • 在一个进程内部运行的多个控制线程，它们彼此分离，但是共享地址空间。
    • 线程可以看做是小型的进程，比进程增多了一个堆栈区。

Inter Process Communication (IPC)

对于共享内存的程序片段称为临界区。无论是进程还是线程，都不可能同时访问临界区，因此要设计方法使得它们无法同时访问临界区，实现互斥。

忙等待

  1. 屏蔽中断。对于某个进程，进入临界区之后屏蔽所有中断，包括时钟中断，这样 CPU 无法强制进行进程切换。

  2. 锁变量。进程间共享锁变量，当有进程进入临界区后加锁，释放临界资源同时释放锁，其他进程访问临界资源时先访问锁变量。缺点在于检测锁变量到设置锁变量的过程不是原子性的，可能有其他进程抢先加锁。

  3. 严格轮转法。连续测试一个变量直到某个值出现，这样的方法称为忙等待(busy waiting)，用于忙等待的锁称为自旋锁(spin lock)。

    严格轮转法是严格的 one-by-one 顺序流程，如果一个进程远远慢于另一个进程，这不是一种好的互斥方法。

  4. Peterson 解法。

  5. TSL 指令。

睡眠与唤醒

  1. 生产者消费者方法

    本质上是有界缓冲区（bounded-buffer）问题。如果缓冲区已满/为空，就不能放入/取出数据，产生睡眠等待。

    这种操作方法的缺点在于 count 的访问未加限制。举例而言，在消费者判断 count == 0 的同时，count 变为 1，那么生产者会调用 wakeUp 通知消费者，然而此时消费者尚未睡眠，因此无法接受这个 wakeUP 信号。这种情况下，生产者和消费者都会卡在 sleep 阶段，而缓存区会满。这种情况的根本原因在于丢失了 wakeUp 信号。

其他方法：信号量、互斥量（mutex）、管程、消息传递、屏障、避免锁等。

iOS 中的多线程之 GCD （Grand Central Dispatch）

基本概念

任务

GCD 中，程序执行就是在进行任务。应用由多个任务组成，而任务的执行需要时间和相应的顺序。任务需按照正确的时间和顺序进行编排。

任务分为两种方式，同步执行（sync）和异步执行（async）。它们的区别是是否等候队列任务执行结束，以及是否具备开启新线程的能力。

  • 同步执行（sync）：

    • 同步添加任务到指定的队列中，在添加的任务执行结束之前，会一直等待，直到队列里面的任务完成之后再继续执行。
    • 只能在当前线程中执行任务，不具备开启新线程的能力。

  • 异步执行（async）：

    • 异步添加任务到指定的队列中，它不会做任何等待，可以继续执行任务。
    • 可以在新的线程中执行任务，具备开启新线程的能力。

异步执行不一定开启新线程，取决于使用的队列类型。

队列（Dispatch Queue）

在 GCD 中有两种队列：串行队列和并发队列。两者都符合 FIFO（先进先出）的原则。两者的主要区别是：执行顺序不同，以及开启线程数不同。

  • 串行队列（Serial Dispatch Queue）：
    • 每次只有一个任务被执行。让任务一个接着一个地执行。（只开启一个线程，一个任务执行完毕后，再执行下一个任务）
  • 并发队列（Concurrent Dispatch Queue）：
    • 可以让多个任务并发（同时）执行。（可以开启多个线程，并且同时执行任务）

使用步骤

GCD 的使用步骤只有两步：

  1. 创建一个队列。
  2. 把任务添加到任务的等待队列，系统根据任务类型执行任务。

swift 使用

队列的声明
  1. 两种队列
  1. 系统主队列和系统全局并发队列

主队列是一个串行队列，它主要处理 UI 相关任务，也可以处理其他类型任务，但为了性能考虑，尽量让主队列执行 UI 相关或少量不耗时间和资源的操作。它通过类属性获取。

全局并发队列，存在 5 个不同的 QoS 级别，可以使用默认优先级，也可以单独指定。

添加队列任务
  1. 同步任务

任务一经提交，就会阻塞当前线程并请求队列立即安排其执行。同步任务直接在当前线程运行，完成后恢复线程原任务。类似于操作系统中 fork 指令由父进程衍生子进程的过程。

多种情况如图所示：

  1. 队列中如果没有任务在执行，那么提交同步任务后，将立即执行该任务，并阻塞线程 thread0，任务完成后再恢复线程 thread0 中被阻塞的任务。

  2. 如果串行队列中有任务在执行，如果该任务又向该队列提交了一个同步任务，将会立即发生死锁。

  3. 如果并行队列中有任务在执行，如果该任务又向该队列提交了一个同步任务，那么当前线程会转而执行新的同步任务，结束后再回到原任务。

  4. 如果队列中有任务在执行，如果该任务向另一个队列提交了一个同步任务，那么当前线程会转而执行新的同步任务，结束后再回到原任务。

  1. 异步任务

任务提交后不会阻塞当前线程，队列会安排另一个线程执行任务。

  1. 迭代任务

迭代任务可以更高效地利用多核性能，它可以利用 CPU 当前所有可用线程进行计算（任务小也可能只用一个线程）。如果一个任务可以分解为多个相似但独立的子任务，那么迭代任务是提高性能最适合的选择。

使用 concurrentPerform 方法执行迭代任务，迭代任务的后续任务需要等待它执行完成才会继续。

任务和队列的特性

  • 串行队列同一时间只会使用同一线程、运行同一任务，并严格按照任务顺序执行。

  • 并行队列同一时间可以使用多个线程、运行多个任务，执行顺序不分先后。

  • 同步任务会阻塞当前线程，并在当前线程执行。

  • 异步任务不会阻塞当前线程，并在与当前线程不同的线程执行。

  • 如何避免死锁：不要在串行或主队列中嵌套执行同步任务。

1. Introduction to DP

基本概念

  • 动态规划(Dynamic Programming) 本质上是将一个问题拆分为几个子问题，通过分别求出子问题的最优解，得到大问题的最优解。
  • 动态规划具有以下两个性质：
  1. 无后效性：「未来与过去无关」。如果给定某一阶段的状态，则在这一阶段以后过程的发展不受这阶段以前各段状态的影响。
  2. 最优子结构：大问题的最优解可以由小问题的最优解推出，这个性质叫做“最优子结构性质”。

例子

给定一个城市的地图，所有的道路都是单行道，而且不会构成环。每条道路都有过路费，问从S点到T点花费的最少费用。

　其中 R 为有路通到 P 的所有的点， 为 R 到 P 的过路费。

思想

DP 的核心思想在于：在可能的解空间内寻找最优解。枚举法得到最大的可能解空间，而DP则是枚举可能的解，即动态规划自带减枝。

设计DP算法的核心在于：

首先，把我们面对的局面表示为x。这一步称为设计状态。
对于状态x，记我们要求出的答案(e.g. 最小费用)为f(x).我们的目标是求出f(T).
找出f(x)与哪些局面有关（记为p），写出一个式子（称为状态转移方程），通过f(p)来推出f(x).

iOS 面试

简述内存管理基本原则

  1. iOS 5.0 之前。采用 MRC 手动管理机制，遵循「谁创建，谁释放；谁引用，谁管理」，创建或引用对象时，需要发生 alloc 、copy、retain 等消息，当释放该对象时需要发送 release 消息，引用计数为 0 时系统释放对象。
  2. iOS 5.0 之后。采用 ARC 自动引用计数，不再需要调用 retain、release、autorelease。采用 strong，weak 关键字：它引用strong和weak关键字，strong修饰的指针变量指向对象时，当指针指向新值或者指针不复存在，相关联的对象就会自动释放，而weak修饰的指针变量指向对象，当对象的拥有者指向新值或者不存在时weak修饰的指针会自动置为nil。

MVC

  1. Controller 和 View 之间通过 DataSource&Delegate

  2. Controller 和 Model 之间通过 Notification&KVO

  3. View 可以设置 action，Controller 设置 target。

Controller是Model和View的协调者，Controller把Model中的数据拿过来给View用。Controller可以直接与Model和View进行通信，而View不能和Controller直接通信。View与Controller通信需要利用代理协议的方式，当有数据更新时，Model也要与Controller进行通信，这个时候就要用Notification和KVO，这个方式就像一个广播一样，MOdel发信号，Controller设置监听接受信号，当有数据更新时就发信号给Controller，Model和View不能直接进行通信，这样会违背MVC设计模式。

Weak

weak表是一个哈希表，kay 指向对象地址，value 指向 weak 指针地址数组。

实现原理：
  1. 初始化时，runtime 调用 objc_initWeak 函数，初始化一个新的 weak 指针指向对象的地址。
  2. 添加引用时，objc_initWeak 函数会调用 objc_storeWeak() 用来更新指针指向，创建对应的弱引用表。
  3. 释放 clearDeealcating 函数，根据对象地址获取所有 weak 指针地址数组，然后遍历数组把其中数据设为 nil，最后把 entry 删除。
